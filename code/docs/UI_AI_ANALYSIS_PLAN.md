# UI: транскрибация, чат и вердикты AI (план)

Цель: подключить UI к уже согласованным WS-событиям (`text_analysis`, `face_analysis`, `audio_analysis`, `analysis_report*`, legacy `emotion`) и убрать ощущение «вечного AI analyzing».

## 1. Источники данных

| Источник | Когда использовать |
|----------|---------------------|
| WebSocket (текущая сессия) | live: partial transcript, эмоции, аудио-фичи, partial report |
| `GET /sessions/:id/analysis/events` | история сегмента, восстановление после F5, отладка |
| `GET /sessions/:id/analysis/report` | итог/последний partial report по сессии |

## 2. Информационная архитектура (рекомендуемый MVP)

1. **Правая колонка «Транскрипт»** (не чат): только поток реплик с таймкодом / `participant_id`, группировка по спикеру. Вход: `text_analysis` (`transcript_partial` → дополняется, `transcript_final` → фиксируется строкой).
2. **Отдельная панель «Чат»** (как сейчас задумано продуктом): только пользовательские сообщения, без смешивания с ASR.
3. **Вердикт / краткий вывод нейросети**:
   - **MVP:** компактная плашка вверху правой колонки или под заголовком встречи: текст из `analysis_report_partial.report.summary` (или выбранное поле контракта).
   - **Расширение:** клик → drawer/modal с полным JSON/структурированным видом.

Альтернатива (можно как фаза 2): **bubble под активным тайлом** — при `text_analysis` для `participant_id` показывать 1–2 строки + «развернуть»; плюсы: контекст у говорящего; минусы: перегруз при нескольких спикерах, нужна чёткая логика «кто сейчас говорит» (VAD/diarization с backend).

## 3. Состояния вместо голого «AI analyzing»

Связать индикатор с реальным пайплайном:

| Состояние UI | Условие (пример) |
|----------------|------------------|
| `Idle` | нет аудио-потока / нет активного спикера |
| `Listening` | идёт захват аудио, ещё нет `text_analysis` |
| `Transcribing` | есть `text_analysis` с `stage=partial` |
| `Analyzing` | есть `audio_analysis` / `face_analysis` без финального отчёта |
| `Verdict ready` | пришёл `analysis_report_partial` или `analysis_report` |

Текст «AI analyzing…» заменить на один из коротких статусов выше или **убрать** и оставить только иконку/тонкую полоску прогресса у соответствующей зоны (транскрипт / вердикт).

## 4. Привязка к существующему коду UI

- Эмоции live: `emeeting-ui/src/pages/VideoMeet.tsx` — совместимость с legacy `emotion` и события `face_analysis`.
- Отчёт: `emeeting-ui/src/pages/Report.tsx` — данные из **`GET /sessions/:id/analysis/report`** (организатор); отображаются расширения заглушки (`meeting_summary`, `participant_tiles`, таймлайны и др.). Маршруты **`/reports`** и **`/reports/:sessionId`**; см. `docs/REPORTS_AND_ANALYTICS_STORAGE.md`.
- **`face_debug`** можно использовать для отладочного оверлея live — сообщения **не** сохраняются в `analysis_event`, см. контракт.

## 5. Фазы внедрения

**Фаза A (1 спринт):** правый рейл «Транскрипт» + обработка `text_analysis` + минимальный статус вместо «AI analyzing».

**Фаза B:** плашка вердикта из `analysis_report_partial` + modal/drawer деталей.

**Фаза C:** отдельный чат (если ещё не выделен) + полировка a11y и mobile (рейл сворачивается).

**Фаза D (опция):** bubble под спикером + diarization в данных (нужен надёжный `participant_id`/speaker id в событиях).

## 6. Контракты (напоминание)

См. `docs/ANALYSIS_WS_CONTRACTS.md`: для новых типов в `payload` обязательны `module`, `version`, `stage`, `trace_id` — UI может группировать и дедуплицировать по `trace_id`.

## 7. Критерии готовности (DoD плана в коде)

- Пользователь видит live-транскрипт без перезагрузки страницы.
- Чат и транскрипт визуально разделены.
- Вердикт partial report виден и раскрывается в детали (правый рейл встречи).
- Состояние загрузки не вводит в заблуждение (нет вечного «analyzing» без событий).
- Организатор открывает отчёт по сессии с **`/reports`** без расхождения с сохранённым телом отчёта на backend.
